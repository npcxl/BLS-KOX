==================================================
一、第一优先级：修复 Replay Protection 执行顺序
=================================

检查：

ReplayProtectionService

目前重点确认 check() 的执行顺序。

错误顺序可能是：

Timestamp
→ Nonce Redis SET NX
→ Signature
→ Idempotency

该顺序存在问题：

Signature 模式下，如果攻击者发送：

合法 Timestamp
+
合法格式 Nonce
+
错误 Signature

如果先写入 Nonce，再验签，则错误签名请求也会消耗 Nonce。

正确顺序：

Timestamp Validation
→ Header Format Validation
→ Signature Validation
→ Nonce Validation
→ Idempotency Validation
→ Controller

请修改为：

mode = off

直接放行。

mode = timestamp

只验证 Timestamp。

mode = nonce

Timestamp
→ Nonce

mode = signature

Timestamp
→ Signature
→ Nonce

注意：

Signature 验证成功之后才能执行 Redis：

SET replayKey 1 NX EX ttl

请保持：

Controller 和 Service 业务层完全不感知 Replay Protection。

二、修复 Nonce TTL 安全窗口
===================

检查规则：

windowSeconds
nonceTtlSeconds

当前如果存在：

windowSeconds = 120
nonceTtlSeconds = 180

可能存在 Timestamp 允许未来时钟偏移时，Nonce 已过期但 Timestamp 仍然有效的边界问题。

增加统一计算方法，例如：

function getEffectiveNonceTtl(
windowSeconds: number,
configuredTtl?: number,
): number {
return Math.max(
configuredTtl ?? 0,
windowSeconds * 2 + 30,
);
}

默认规则建议：

windowSeconds: 120
nonceTtlSeconds: 300

高危接口可以使用更大的 TTL。

不要在多个地方重复计算 TTL。

封装统一方法。

三、修复 Idempotency 并发竞争
=====================

重点检查：

checkIdempotent()

当前如果逻辑类似：

const current = await redis.get(key);

if (!current) {
await redis.set(
key,
"processing",
"EX",
ttl,
"NX",
);

return {
status: "new",
};
}

这是错误的。

原因：

请求 A：
GET → null

请求 B：
GET → null

A：
SET NX → OK

B：
SET NX → null

如果代码不检查 SET NX 返回结果：

A 和 B 都可能返回 new。

必须修复。

推荐：

const lockResult = await redis.set(
key,
processingValue,
"EX",
ttlSeconds,
"NX",
);

if (lockResult === "OK") {
return {
status: "new",
};
}

const current = await redis.get(key);

if (current === "processing") {
throw IDEMPOTENCY_PROCESSING;
}

if (current) {
return {
status: "completed",
cachedResponse: current,
};
}

请进一步考虑：

SET NX 后状态读取期间发生变化的竞争问题。

优先使用：

Redis Lua Script

或者设计清晰的原子状态机。

不要：

GET
→ SET

这种非原子方式决定首次请求资格。

五、处理业务异常后的 Idempotency processing 状态
====================================

检查下面情况：

第一次请求：

Idempotency Key 占位 processing

然后：

Controller 抛异常

如果 processing 一直保留到 TTL 过期，则客户端很长时间不能正常重试。

请设计：

try {
await next();

success：
completed
} catch (error) {
release processing lock
或记录 failed 状态

throw error;
}

需要考虑：

1. 4xx 是否允许重试
2. 5xx 是否释放锁
3. 超时如何处理
4. Worker 崩溃时依赖 TTL 自动恢复

推荐最简单策略：

2xx：
completed

明确不可重试业务错误：
可以缓存业务结果

5xx：
删除 processing Key，允许客户端使用相同 Idempotency-Key 重新请求

请根据当前项目错误体系适配。
