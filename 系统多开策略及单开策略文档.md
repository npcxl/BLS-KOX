# 系统多开登录与单点登录实现说明

## 1. 先说结论

这个文档给开发者看，重点不是讲概念，而是讲**怎么做**。

在 BLS 里，登录态一般是这样处理的：

- 登录成功后签发 `token`
- `token` 存到 `Redis`
- 前端把 `token` 带在请求头里
- 后端每次请求先校验 `token` 是否存在、是否过期、是否被踢下线

所以这里真正要讨论的是两种实现方式：

- **多开登录**：同一个账号允许生成多个 token，同时有效
- **单点登录 / 单会话登录**：同一个账号只保留一个有效 token，新登录会覆盖旧登录，或通过统一认证中心共享登录态

如果你们内部说的是“单开”，开发上更准确的说法其实是：**单会话登录**。如果还涉及多个系统共用一次登录，那就是 **单点登录（SSO）**。

## 2. 当前系统的登录态模型

通常可以理解成下面这套链路：

1. 用户提交账号密码
2. 后端校验通过后生成 `token`
3. `token` 写入 `Redis`
4. 前端保存 `token`
5. 后续请求携带 `token`
6. 后端从 `Redis` 查 `token` 是否有效

如果 `Redis` 里查不到这个 token，说明：

- token 过期了
- token 被主动删除了
- token 被新登录覆盖了
- 用户被强制下线了

所以，**Redis 里的 token 数据就是登录态的核心**。

## 3. 多开登录怎么实现

### 3.1 核心目标

同一个账号可以同时在多个设备、多个浏览器、多个窗口里登录，而且互不影响。

### 3.2 实现方式

最简单的做法是：

- 每次登录都生成一个新的 `token`
- `token` 作为独立 key 存入 `Redis`
- 不做账号级别的互斥限制
- 只要 token 没过期、没被删除，就算有效

### 3.3 Redis 存储方式

可以按 token 维度存：

- `login:token:{token} -> userId / 用户信息`
- 设置过期时间，比如 2 小时、8 小时、24 小时

这样同一个账号多次登录，会在 Redis 里同时存在多个 token：

- `login:token:aaa111`
- `login:token:bbb222`
- `login:token:ccc333`

每个 token 都能正常访问系统，这就是多开。

### 3.4 接口校验逻辑

请求进来后：

1. 从请求头拿到 token
2. 去 Redis 查 `login:token:{token}`
3. 查到了就放行
4. 查不到就返回未登录

### 3.5 适用场景

- 允许用户在手机和电脑同时在线
- 允许同一账号多处协同操作
- 业务对“只能一个人在线”没有要求

## 4. 单点登录怎么实现

### 4.1 核心目标

用户只登录一次，就能访问多个系统或多个子系统。

### 4.2 开发实现思路

如果是多个系统共用一套登录，常见做法是：

- 统一认证中心负责登录
- 登录后发一个统一票据 / token
- 各业务系统只负责验票，不自己单独登录
- 退出时统一清理认证状态

### 4.3 在 BLS 里的落地方式

如果 BLS 只有单体系统，不涉及多个子系统，其实更多是做“单会话控制”，而不是完整的 SSO。

如果后续有：

- `bls-admin`
- `bls-app`
- `bls-portal`
- 其他业务子系统

那就可以上 SSO。

### 4.4 技术重点

- 统一认证入口
- 统一 token 颁发
- 统一退出登录
- 子系统共享登录态
- token 或票据需要可追踪、可失效

## 5. 单会话登录怎么实现

这个是你们现在最可能要的版本。

### 5.1 核心目标

同一个账号同一时间只允许一个有效登录态。

### 5.2 最常见实现

登录时先查这个账号是否已经登录：

- 如果没有登录：直接生成新 token，写入 Redis
- 如果已经登录：删除旧 token，再写入新 token

也就是“后登录覆盖前登录”。

### 5.3 Redis 设计方式

建议同时保存两类数据：

#### 方式一：只存 token

- `login:token:{token} -> userId`

这种方式适合多开，不适合做单会话，因为你不知道一个用户当前有几个 token。

#### 方式二：同时存用户和 token 的映射

- `login:token:{token} -> userId`
- `login:user:{userId} -> token`

这种方式最常用。

登录时流程：

1. 先查 `login:user:{userId}`
2. 如果存在旧 token，删掉旧 token 对应的 `login:token:{oldToken}`
3. 写入新的 `login:token:{newToken}`
4. 更新 `login:user:{userId} -> newToken`

这样就能保证同一个账号只保留一个 token。

### 5.4 请求校验逻辑

请求进入后：

1. 从请求头取 token
2. 去 Redis 查 `login:token:{token}`
3. 再根据 userId 查 `login:user:{userId}`
4. 如果当前 token 不是最新 token，直接判定失效

这样即使旧 token 没过期，只要被新登录覆盖，也会立刻失效。

### 5.5 强制下线逻辑

如果后台要支持“踢下线”，直接做：

- 删除 `login:token:{token}`
- 删除 `login:user:{userId}`

或者只删除 `login:user:{userId}`，再在请求时通过最新 token 判断失效。

### 5.6 适用场景

- 安全要求高
- 不允许账号共享
- 希望同一账号只能一个人在线
- 管理后台、运营后台、财务相关系统

## 6. 多开和单会话的区别

| 项目 | 多开登录 | 单会话登录 | 单点登录 |
|---|---|---|---|
| Redis 里是否可以同时存在多个 token | 可以 | 不建议 | 可以 |
| 同一账号是否允许多处同时在线 | 允许 | 不允许 | 取决于会话策略 |
| 重点解决的问题 | 多终端同时在线 | 账号互斥 | 多系统统一登录 |
| 是否需要统一认证中心 | 不一定 | 不一定 | 通常需要 |
| 实现复杂度 | 低 | 中 | 高 |

## 7. 开发上推荐怎么做

如果你们现在是基于 `Redis + token` 的方案，我建议这样拆：

### 7.1 如果要多开

- token 独立生成
- Redis 只按 token 存
- 不做账号互斥
- 前端拿哪个 token，就用哪个 token

### 7.2 如果要单会话

- `Redis` 同时保存 `userId -> token`
- 新登录覆盖旧 token
- 校验时同时判断 token 和 userId 绑定关系
- 支持强制下线旧会话

### 7.3 如果要单点登录

- 单独做认证中心
- 各系统接入统一登录
- token 或票据统一颁发、统一校验、统一退出

### 7.2 选择单点登录

如果系统由多个子系统组成，且希望统一身份认证入口，应采用单点登录。

### 7.3 选择单会话登录

如果系统安全要求高，且需要防止账号共享和多端冲突，应采用单会话登录。

## 8. 风险与注意事项

- **多开登录**可能带来账号共享和操作冲突风险
- **单点登录**需要处理统一认证、退出同步、票据安全等问题
- **单会话登录**可能影响多端协同和用户体验
