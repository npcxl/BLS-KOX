# 动态接口生成文档

BLS 后端支持三种方式定义 API 接口，从零代码到完全自定义，按需使用。

---

## 一、三种接口定义方式速览

| 方式 | 代码量 | 适用场景 | 示例 |
|------|--------|----------|------|
| **纯配置模式** | 5 行 | 标准单表 CRUD | `export const config = { table:'sys_order', pkField:'order_id', searchFields:['order_name'] }` |
| **混合模式** | 10-30 行 | 自定义 list 接口 + 其余 CRUD 自动生成 | 导出 `config` + 导出 `Router` |
| **纯自定义模式** | 不限 | 完全自定义所有接口 | 只导出 `Router`（不导出 `config`） |

---

## 二、纯配置模式（最简）

只需在 `index.ts` 中导出一个 `config` 对象，**无需写任何路由代码**。

### 2.1 文件结构

```
bls-server/src/api/
└── business/
    └── orders/
        └── index.ts    ← 只放一个 config
```

### 2.2 代码

```ts
// bls-server/src/api/business/orders/index.ts

export const config = {
  table: 'orders',               // 数据库表名
  pkField: 'id',                 // 主键字段
  searchFields: ['order_no', 'customer_name'],  // 关键字搜索字段
  name: '订单',                   // 模块中文名
  permPrefix: 'business:order',  // 权限前缀
};
```

### 2.3 自动生成的接口

| 方法 | 完整路径 | 权限 | 说明 |
|------|----------|------|------|
| `GET` | `/api/business/orders/list` | `business:order:list` | 分页查询 |
| `POST` | `/api/business/orders/add` | `business:order:add` | 新增 |
| `PUT` | `/api/business/orders/edit` | `business:order:edit` | 修改 |
| `DELETE` | `/api/business/orders/remove` | `business:order:remove` | 批量删除 |
| `PUT` | `/api/business/orders/status` | `business:order:status` | 状态切换 |

> **路由前缀自动推导**：文件夹路径 `api/business/orders/` → 自动得到前缀 `/business/orders`。

### 2.4 完整配置项

```ts
export interface CrudModuleConfig {
  /** 路由前缀，不填则从文件夹路径自动推导 */
  prefix?: string;
  /** 数据库表名（必填） */
  table: string;
  /** 主键字段名，snake_case（必填），如 'role_id' */
  pkField: string;
  /** 租户字段名，默认 'tenant_id' */
  tenantField?: string;
  /** 状态字段名，默认 'status' */
  statusField?: string;
  /** 是否软删除，默认 true */
  softDelete?: boolean;
  /** 关键字搜索字段，snake_case */
  searchFields?: string[];
  /** 模块中文名 */
  name?: string;
  /** 权限前缀，如 'system:role' */
  permPrefix?: string;
  /** Zod 校验 Schema */
  schema?: { create?: z.ZodType; update?: z.ZodType };
}
```

### 2.5 各接口行为说明

#### GET /list

```http
GET /api/business/orders/list?pageNum=1&pageSize=10&keyword=测试&status=0
```

| 参数 | 说明 |
|------|------|
| `pageNum` | 页码，默认 1 |
| `pageSize` | 每页条数，默认 10，最大 100 |
| `keyword` | 模糊搜索，在 `searchFields` 所有字段上做 `LIKE %keyword%` |
| 任意其他参数 | 精确过滤，如 `status=0` → `WHERE status = '0'` |

**内置行为**：
- 自动添加租户过滤：`WHERE tenant_id = '当前租户ID'`
- `softDelete=true` 时自动添加：`WHERE deleted = 0`
- 返回字段自动 `snake_case` → `camelCase`

**响应格式**：
```json
{
  "code": 200,
  "data": [{ "orderId": "123", "orderNo": "ORD001", ... }],
  "total": 100
}
```

#### POST /add

```http
POST /api/business/orders/add
Content-Type: application/json

{ "orderNo": "ORD001", "customerName": "张三", "status": "0" }
```

**内置行为**：
- 主键未传时自动生成 Snowflake ID
- 租户 ID 自动注入（取请求体中的值，否则用当前用户租户）
- `softDelete=true` 时自动补 `deleted=0`
- 请求体字段自动 `camelCase` → `snake_case` 后写入数据库

#### PUT /edit

```http
PUT /api/business/orders/edit
Content-Type: application/json

{ "id": "123", "customerName": "李四", "status": "1" }
```

**内置行为**：
- 主键兼容两种写法：`{ "id": "123" }` 和 `{ "orderId": "123" }` 均可
- 自动移除主键字段（防止 update 主键）
- 自动租户隔离 + 软删除过滤（防止跨租户修改或修改已删除记录）

#### DELETE /remove

```http
DELETE /api/business/orders/remove
Content-Type: application/json

{ "ids": ["123", "456"] }
```

- 支持单个或批量删除
- `softDelete=true`：`UPDATE SET deleted=1`（软删除）
- `softDelete=false`：`DELETE FROM`（物理删除）
- 自动租户过滤（防止跨租户删除）

#### PUT /status

```http
PUT /api/business/orders/status
Content-Type: application/json

{ "id": "123", "status": "1" }
```

- 简单状态切换（如启用/停用）
- 自动租户隔离 + 软删除过滤

### 2.6 snake_case ↔ camelCase 自动转换

整个系统在以下位置自动处理命名风格转换：

| 方向 | 位置 | 说明 |
|------|------|------|
| 请求体 → 数据库 | `crud.ts` `toSnake()` | `customerName` → `customer_name` |
| 数据库 → 响应体 | `router.ts` `wrapCamel()` | `customer_name` → `customerName` |
| query 参数 → SQL | `crud.ts` `/list` | `customerName` → `customer_name` |

前端只需使用 camelCase，无需关心数据库的 snake_case。

## 三、混合模式（自定义 + 自动）

当你需要自定义某些接口（如 `GET /list` 返回树形数据），又想保留其余 CRUD 接口自动生成时使用。

### 3.1 文件结构

```
bls-server/src/api/
└── system/
    └── dept/
        └── index.ts    ← 同时导出 Router + config
```

### 3.2 代码

```ts
// bls-server/src/api/system/dept/index.ts
import Router from 'koa-router';
import { getDb, jwtAuth, hasPerm } from '../../../core';

const router = new Router({ prefix: '/system/dept' });
const T = 'sys_dept';

// ======== 自定义路由（优先匹配） ========

// 覆盖 GET /list：返回树形数据
router.get('/list', jwtAuth(), hasPerm('system:dept:list'), async (ctx) => {
  const rows = await (await getDb()).selectFrom(T).selectAll()
    .where('deleted', '=', 0).orderBy('sort_num', 'asc').execute();
  ctx.body = { code: 200, data: buildTree(rows) };
});

// 自定义 POST /add（例如需要特殊校验）
router.post('/add', jwtAuth(), hasPerm('system:dept:add'), async (ctx) => {
  // ... 自定义逻辑
});

// ======== 导出默认 Router ========
export default router;

// ======== 导出 config（其余 CRUD 自动生成） ========
export const config = {
  table: 'sys_dept',
  pkField: 'dept_id',
  searchFields: ['dept_name'],
  name: '部门',
  permPrefix: 'system:dept',
};
```

### 3.3 匹配优先级

```
自定义 Router 中的路由  →  先匹配（高优先级）
defineCrudModule 的 CRUD →  后匹配（兜底）
```

即：你覆盖了 `/list`，就用你的；你没覆盖的 `/remove`、`/status` 等，由 `config` 自动生成。

## 四、纯自定义模式（完全控制）

当你不需要任何标准 CRUD 时，只导出 `Router`，不导出 `config`。

### 4.1 代码

```ts
// bls-server/src/api/custom/dashboard/index.ts
import Router from 'koa-router';

const router = new Router({ prefix: '/custom/dashboard' });

router.get('/stats', async (ctx) => {
  // 完全自定义逻辑
  ctx.body = { code: 200, data: { totalUsers: 100, totalOrders: 500 } };
});

router.post('/refresh', async (ctx) => {
  // ...
  ctx.body = { code: 200, message: '刷新成功' };
});

export default router;
// 不导出 config → 纯自定义模式，不生成标准 CRUD
```

## 五、函数导出模式

除了导出 `config` 和 `Router`，你还可以**直接导出命名函数**，系统会自动将其注册为 API 路由。

### 5.1 代码

```ts
// bls-server/src/api/system/product/index.ts

// 函数 1：getList → GET /list（需要登录）
export async function getList(ctx) {
  const rows = await (await getDb()).selectFrom('sys_product').selectAll().execute();
  return rows;  // 直接 return 数据，系统自动包装为 { code: 200, data: ... }
}

// 函数 2：exportData → POST /export-data（需要登录）
export async function exportData(ctx) {
  const body = ctx.request.body;
  // ...
  return { url: 'https://...' };
}

// 函数 3：publicInfo → GET /public-info（无需登录，public 开头）
export async function publicInfo(ctx) {
  return { name: 'BLS', version: '1.0' };
}

// 同时还可以导出 config，让标准 CRUD 也生效
export const config = {
  table: 'sys_product',
  pkField: 'product_id',
  searchFields: ['product_name'],
  name: '产品',
  permPrefix: 'system:product',
};
```

### 5.2 函数命名规则

| 函数名 | HTTP 方法 | 路径 | 是否需要登录 |
|--------|-----------|------|-------------|
| `getList` | `GET` | `/list` | ✅ 是 |
| `addUser` | `POST` | `/add-user` | ✅ 是 |
| `editRole` | `PUT` | `/edit-role` | ✅ 是 |
| `removeItem` | `DELETE` | `/remove-item` | ✅ 是 |
| `saveData` | `POST` | `/save-data` | ✅ 是 |
| `createOrder` | `POST` | `/create-order` | ✅ 是 |
| `updateConfig` | `PUT` | `/update-config` | ✅ 是 |
| `deleteLog` | `DELETE` | `/delete-log` | ✅ 是 |
| `publicHealth` | `GET` | `/public-health` | ❌ 无需 |
| `publicStatus` | `GET` | `/public-status` | ❌ 无需 |
| `login`、`logout`、`refresh` | `POST` | `/login` 等 | ❌ 无需 |

**命名规则**：
- 大驼峰字母 → 连字符：`getList` → `/list`，`addUser` → `/add-user`
- `add*` / `create*` / `save*` → POST
- `edit*` / `update*` → PUT
- `delete*` / `remove*` → DELETE
- 其他 → GET
- `public*` / `Public*` / `login` / `logout` / `refresh` → 无需 jwtAuth

**被跳过的函数名**：
- 大写字母开头的函数（如 `Service`、`ConfigService`）→ 不注册为路由
- `config`、`default`、`Router` → 不注册为路由

## 六、文件组织规范

### 6.1 自动扫描机制

```
bls-server/src/api/
├── system/
│   ├── user/
│   │   └── index.ts       ← 扫描到，注册为 /system/user
│   ├── role/
│   │   └── index.ts       ← 扫描到，注册为 /system/role
│   └── dept/
│       └── index.ts       ← 扫描到，注册为 /system/dept
└── business/
    └── orders/
        └── index.ts       ← 扫描到，注册为 /business/orders
```

**核心规则**：
1. 只扫描各文件夹下的 `index.ts` / `index.js`
2. 路由前缀 = 文件夹相对于 `api/` 的路径：`api/system/user/` → `/system/user`
3. 不递归扫描子文件夹（除非文件夹下没有 `index.ts`）

### 6.2 会被跳过的文件名

以下文件名**不会**被注册为路由（防止工具文件被误注册）：

| 文件名 | 原因 |
|--------|------|
| `model.ts` | 数据模型文件 |
| `*.routes.ts` | 路由定义文件 |
| `*.controller.ts` | 控制器文件 |
| `*.service.ts` | 服务文件 |
| `*.repository.ts` | 数据访问文件 |
| `excel.*.ts` | Excel 工具文件 |

## 七、权限控制

### 7.1 config 模式的权限

`defineCrudModule` 根据 `permPrefix` 自动生成权限：

| 接口 | 自动权限 |
|------|----------|
| `GET /list` | `{permPrefix}:list` |
| `POST /add` | `{permPrefix}:add` |
| `PUT /edit` | `{permPrefix}:edit` |
| `DELETE /remove` | `{permPrefix}:remove` |
| `PUT /status` | `{permPrefix}:status` |

示例：`permPrefix: 'system:role'` → 自动生成 `system:role:list` 等。

### 7.2 自定义 Router 的权限

自定义 Router 中需要手动指定权限：

```ts
router.get('/list', jwtAuth(), hasPerm('system:dept:list'), async (ctx) => { ... });
router.post('/add', jwtAuth(), hasPerm('system:dept:add'), async (ctx) => { ... });
```

### 7.3 无需登录的接口

以下情况不需要传递 Token：
- `public*` / `Public*` 开头的函数
- 名为 `login` / `logout` / `refresh` 的函数

Router 模式下，不添加 `jwtAuth()` 中间件即可：

```ts
router.get('/public-info', async (ctx) => { ... });  // 无需登录
```

## 八、完整示例：从零新建一个模块

假设要新建一个「标签管理」模块：

### 第 1 步：创建数据库表

```sql
CREATE TABLE `sys_tag` (
  `tag_id` varchar(32) NOT NULL,
  `tenant_id` varchar(32) NOT NULL DEFAULT '000000',
  `tag_name` varchar(100) NOT NULL,
  `color` varchar(20) DEFAULT NULL,
  `status` char(1) NOT NULL DEFAULT '0',
  `deleted` tinyint NOT NULL DEFAULT '0',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`tag_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 第 2 步：创建后端文件

```ts
// bls-server/src/api/business/tags/index.ts

export const config = {
  table: 'sys_tag',
  pkField: 'tag_id',
  searchFields: ['tag_name'],
  name: '标签',
  permPrefix: 'business:tag',
};
```

就这两步，接口全部自动生成

### 第 3 步：创建前端页面

```tsx
// bls-admin/src/pages/business/tags/index.tsx
import { usePageConfig } from '@/hooks/usePageConfig';
import CrudTablePage from '@/components/CrudTablePage';

export type TagRecord = {
  tagId: string;
  tagName: string;
  color?: string;
  status: string;
  createTime?: string;
};

export default function TagPage() {
  const { proColumns } = usePageConfig('business_tag');

  const formColumns: ProFormColumnsType<TagRecord>[] = [
    { title: '标签名称', dataIndex: 'tagName', formItemProps: { rules: [{ required: true }] } },
    { title: '颜色', dataIndex: 'color' },
    { title: '状态', dataIndex: 'status', valueType: 'select', initialValue: '0' },
  ];

  return (
    <CrudTablePage<TagRecord>
      title="标签管理"
      rowKey="tagId"
      resource={{ basePath: '/api/business/tags' }}
      columns={proColumns}
      formColumns={formColumns}
      permissions={{
        create: 'business:tag:add',
        edit: 'business:tag:edit',
        remove: 'business:tag:remove',
      }}
    />
  );
}
```

### 最终效果

| 层 | 代码行数 | 说明 |
|----|---------|------|
| 数据库 | 12 行 SQL | 建表 |
| 后端 | 7 行 TS config | 零路由代码 |
| 前端 | 25 行 TSX | 一个 CrudTablePage |
| **总计** | **44 行** | 完整的增删改查管理模块 |

## 九、常见问题

### Q1：config 模式下如何跳过租户过滤？

设置 `tenantField` 为一个不存在的字段名：

```ts
export const config = {
  table: 'sys_menu',
  pkField: 'menu_id',
  tenantField: undefined,  // 菜单无租户隔离
};
```

但 `getCurrentTenantId()` 返回 falsy 值时不会生效（需要确保返回 null/undefined）。

### Q2：如何额外导出公共方法？

函数导出模式下，导出的所有函数都会自动注册为路由。如果只想导出工具函数而不注册为路由，有大写字母开头即可：

```ts
// 会注册为路由：POST /add-tag
export async function addTag(ctx) { ... }

// 不会注册为路由（大写开头）
export async function ValidateTagName(name: string) { ... }
```

### Q3：如何查看当前注册了哪些路由？

所有路由注册日志在终端输出，也可以在 `/api/system/page-config` 页面查看已配置的页面（路径对应路由）。

### Q4：请求体中的字段名应该用 snake_case 还是 camelCase？

**全部使用 camelCase**。后端自动转换：

```
前端发送: { tagName: "xxx" }
后端接收: { tagName: "xxx" }
数据库写入: { tag_name: "xxx" }
数据库返回: { tag_name: "xxx", ... }
前端收到: { tagName: "xxx", ... }
```

## 十、动态列配置

BLS 支持**运行时动态配置页面列**，无需改代码即可调整表格列的显示、搜索、编辑等属性。

### 10.1 概述

传统方式：
```
硬编码 columns → 要改列顺序 → 改代码 → 重新编译 → 部署
```

动态列配置：
```
数据库 sys_page_column_config → usePageConfig() 自动加载 → 页面实时生效
```

**核心理念**：页面列定义从代码中剥离，存放在数据库里，通过管理页面实时调整。

### 10.2 数据表结构

**`sys_page_config`（页面配置）**：

```sql
CREATE TABLE `sys_page_config` (
  `page_config_id` varchar(32) NOT NULL COMMENT '页面配置ID',
  `page_code` varchar(100) NOT NULL COMMENT '页面编码（如 system_user）',
  `page_name` varchar(100) NOT NULL COMMENT '页面名称（如 用户管理）',
  `enabled` tinyint(1) NOT NULL DEFAULT '1' COMMENT '是否启用',
  `sort` int NOT NULL DEFAULT '0' COMMENT '排序',
  `tenant_id` varchar(32) NOT NULL DEFAULT '000000' COMMENT '租户ID',
  `remark` varchar(500) DEFAULT NULL COMMENT '备注',
  PRIMARY KEY (`page_config_id`),
  UNIQUE KEY `uk_page_code_tenant` (`tenant_id`,`page_code`)
);
```

**`sys_page_column_config`（列配置）**：

```sql
CREATE TABLE `sys_page_column_config` (
  `column_id` varchar(32) NOT NULL COMMENT '列配置ID',
  `page_code` varchar(100) NOT NULL COMMENT '所属页面编码',
  `data_index` varchar(100) NOT NULL COMMENT '字段标识（camelCase，如 roleName）',
  `title` varchar(100) NOT NULL COMMENT '列标题（如 角色名称）',
  `order_num` int NOT NULL DEFAULT '0' COMMENT '排序（数值越小越靠前）',
  `visible` tinyint(1) NOT NULL DEFAULT '1' COMMENT '是否在表格显示',
  `searchable` tinyint(1) NOT NULL DEFAULT '0' COMMENT '是否可搜索',
  `editable` tinyint(1) NOT NULL DEFAULT '0' COMMENT '是否可编辑（表单中出现）',
  `copyable` tinyint(1) NOT NULL DEFAULT '0' COMMENT '是否可复制',
  `ellipsis` tinyint(1) NOT NULL DEFAULT '0' COMMENT '是否超出省略',
  `value_type` varchar(50) DEFAULT NULL COMMENT '值类型：text/select/dateTime/digit/textarea/switch/treeSelect',
  `value_enum_code` varchar(100) DEFAULT NULL COMMENT '字典编码（如 sys_status）',
  `placeholder` varchar(200) DEFAULT NULL COMMENT '占位提示文本',
  `required` tinyint(1) NOT NULL DEFAULT '0' COMMENT '是否必填',
  PRIMARY KEY (`column_id`),
  KEY `idx_col_page_code` (`page_code`)
);
```

### 10.3 前端使用：`usePageConfig` Hook

**文件**：`bls-admin/src/hooks/usePageConfig.ts`

```tsx
import { usePageConfig } from '@/hooks/usePageConfig';

function UserPage() {
  const { proColumns, formColumns, loading } = usePageConfig('system_user');

  return (
    <CrudTablePage
      title="用户管理"
      rowKey="userId"
      resource={{ basePath: '/api/system/user' }}
      columns={proColumns}   // ← 动态生成的 ProTable 列
      formColumns={formColumns}  // ← 动态生成的表单列
    />
  );
}
```

**Hook 内部流程**：

```
usePageConfig('system_user')
  → GET /api/system/page-config/page/system_user/columns
  → 读取所有列配置（visible=true 生成列表列，editable=true 生成表单列）
  → 自动加载字典数据（value_enum_code → valueEnum）
  → 返回 proColumns（ProTable 格式）+ formColumns（ProForm 格式）
```

**返回值**：

| 属性 | 类型 | 说明 |
|------|------|------|
| `columns` | `PageColumnConfigRecord[]` | 原始列配置数据 |
| `proColumns` | `ProColumns[]` | ProTable 列定义（过滤 `visible=false`，自动挂字典 valueEnum） |
| `formColumns` | `ProFormColumnsType[]` | 表单列定义（过滤 `editable=false`，必填字段自动加校验规则） |
| `loading` | `boolean` | 加载状态 |

**列字段映射关系**：

| 数据库字段 `sys_page_column_config` | ProColumns 属性 | 说明 |
|--------------------------------------|-----------------|------|
| `title` | `title` | 列标题 |
| `data_index` | `dataIndex` | 数据字段 |
| `visible` | （过滤条件） | `false` 时从表格中隐藏 |
| `searchable` | `search: true/false` | 是否出现在搜索栏 |
| `ellipsis` | `ellipsis` | 超出省略 |
| `copyable` | `copyable` | 可复制 |
| `editable` | （过滤条件） | `false` 时从表单中隐藏 |
| `value_type` | `valueType` | 值类型（text/select/dateTime 等） |
| `value_enum_code` | `valueEnum` | 自动加载字典数据 |
| `required` | `formItemProps.rules` | 必填校验 |
| `order_num` | （排序依据） | 决定列的顺序 |
| `placeholder` | `fieldProps.placeholder` | 占位提示 |

### 10.4 后端搜索联动（动态列 → 精确搜索）

**文件**：各模块的 `index.ts`（如 `user/index.ts`、`role/index.ts`）

列表接口从 `sys_page_column_config` 动态读取 `searchable=1` 的字段，自动支持精确过滤：

```ts
// bls-server/src/api/system/role/index.ts (GET /list)
router.get('/list', jwtAuth(), hasPerm('system:role:list'), async (ctx) => {
  const db = await getDb();
  const q = ctx.query as any;

  let b = db.selectFrom('sys_role').selectAll().where('deleted', '=', 0);

  // 从动态列配置加载 searchable=1 的字段
  const searchCols = await db.selectFrom('sys_page_column_config')
    .select('data_index')
    .where('page_code', '=', 'system_role')
    .where('searchable', '=', 1)
    .where('deleted', '=', 0)
    .execute();

  const searchFields = searchCols.map((c: any) =>
    c.data_index.replace(/[A-Z]/g, (m: string) => '_' + m.toLowerCase())
  );

  // keyword 模糊搜索：OR LIKE 所有 searchable 字段
  if (q.keyword) {
    b = b.where((eb: any) =>
      eb.or(searchFields.map((f: string) => eb(f, 'like', `%${q.keyword}%`)))
    );
  }

  // searchable=1 的字段自动支持 = 精确过滤（如 ?status=0）
  for (const c of searchCols) {
    const field = c.data_index;
    if (q[field] !== undefined && q[field] !== '' && q[field] !== null) {
      b = b.where(
        field.replace(/[A-Z]/g, (m: string) => '_' + m.toLowerCase()),
        '=',
        String(q[field])
      );
    }
  }

  // ... 分页计数 + 查询
});
```

**效果**：页面配置人员只需在管理页面勾选 `searchable`，后端自动支持该字段的模糊搜索和精确过滤，无需改后端代码。

### 10.8 常见用法

**调整列顺序**：改 `order_num` → 保存 → 刷新

**隐藏某列**：取消勾选 `visible` → 保存 → 刷新列消失

**让某个字段可搜索**：勾选 `searchable` → 保存 → 后端自动支持精确过滤和模糊搜索

**让某个字段在表单中显示**：勾选 `editable` → 保存 → 新增/编辑弹窗中出现该字段

**关联字典**：设置 `value_enum_code` = `sys_status` → 自动渲染为 Tag + 搜索下拉选择
